# ユーザーストーリー

## ペルソナ

### P-001: ローカル LLM 評価担当者

複数のローカル LLM サーバを比較し、評価対象、評価条件、出力をプロジェクト内で再現可能にしたい。

### P-002: CLI 開発者

評価フレームワーク本体を単体テストでは実行せず、設定解決、コマンド生成、結果保存を小さく検証したい。

### P-003: 日本語 QA データセット管理者

日本語を含む自作データセットを破損させず、評価設定と成果物で扱いたい。

## ストーリー

### US-001: suite を検証する

ローカル LLM 評価担当者として、評価実行前に suite YAML の参照と必須項目を検証したい。なぜならば、重い評価を始める前に設定ミスを見つけたいからだ。

### US-002: 評価予定を dry-run で確認する

ローカル LLM 評価担当者として、外部評価フレームワークを起動せずに model x benchmark の実行予定コマンドを確認したい。なぜならば、ローカルサーバや eval dependency を消費せずにレビューしたいからだ。

### US-003: 複数ジョブを suite から展開して実行する

ローカル LLM 評価担当者として、suite に書いた model と benchmark の組み合わせを自動で評価ジョブへ展開したい。なぜならば、同じ評価バッチを再現可能に実行したいからだ。

### US-004: 評価フレームワークごとのコマンドを隠蔽する

ローカル LLM 評価担当者として、lm-evaluation-harness と OpenCompass のコマンド差分を YAML から吸収したい。なぜならば、runner ごとの細かい起動引数を毎回手で組み立てずに済ませたいからだ。

### US-005: 実行結果の summary を残す

ローカル LLM 評価担当者として、各ジョブの status、exit code、raw 出力パス、実行コマンドを summary として残したい。なぜならば、失敗したジョブと再確認すべき出力を追跡したいからだ。

### US-006: 外部依存なしに単体テストする

CLI 開発者として、重い評価フレームワークやローカル LLM サーバを起動せずに主要ロジックを検証したい。なぜならば、通常の開発サイクルで速く安全に変更したいからだ。

### US-007: 日本語データを壊さない

日本語 QA データセット管理者として、日本語を含む JSON と出力を UTF-8 のまま扱いたい。なぜならば、質問文や回答がエスケープや文字化けで読みづらくならないようにしたいからだ。

## ストーリーマップ

1. 設定を作る: model YAML、benchmark YAML、suite YAML を作成する。
2. 検証する: `validate` で必須項目、runner 名、参照ファイルを確認する。
3. 実行予定を確認する: `run --dry-run` で生成コマンドと出力構造を確認する。
4. 実行する: runner adapter が外部評価フレームワーク向けコマンドを生成し、順次実行する。
5. 結果を確認する: `runs/<timestamp>-<suite>/summary.json` と raw 出力を見る。
